Using ARIA

Attributes that polyfill missing role, name, and state information for screen readers

  • ARIA 1.1 (W3C Recommendation):
    w3.org/tr/wai-aria-1.1/
  • Rules for Using ARIA (W3C Note):
    w3.org/TR/using-aria/

  • tablist, tab, tabpanel
  • checkbox, radio, textbox

  • aria-invalid, aria-required

The functionality

var button = document.getElementById('button');

function doSomething(event) {        
    if (button.getAttribute('aria-pressed') == 'false') {
        button.setAttribute('aria-pressed', 'true');
    }
    else {
        button.setAttribute('aria-pressed', 'false');
    }
}

Future

Accessibility Object Model (AOM)

An experimental JavaScript accessibility API:
github.com/WICG/aom

Phase 1A: ARIA reflection

Instead of:

button.setAttribute('role', 'button');
button.setAttribute('aria-pressed', 'false');

It would be:

button.role = 'button';
button.ariaPressed = 'false';

Phase 1B: Custom Element semantics

Authors will be able to:

  • Set immutable default semantics by setting properties through the ElementDefinitionOptions object
  • Modify semantics in response to user interaction using the ElementInternals object

Phase 2: User action events

Authors will be able to respond to new types of input event, including:

  • actionIncrement and actionDecrement
  • actionDismiss
  • actionScrollpageUp and actionScrollPageDown

Phase 3: Virtual accessibility nodes

Will allow authors to create virtual accessibility tree nodes that do not map to the DOM

Phase 4: Computed accessibility tree

Will allow authors to:

  • Access all computed properties for nodes in the accessibility tree
  • Walk the computed accessibility tree (including virtual nodes)

Laws of unintended consequences

There is a trade-off between accessibility and privacy
